## 这份文档的作用

这份文档不是技术设计细节，而是“不要跑偏”的意图备忘录。

项目本质是一个 Codex Desktop 上下文代理：它拦截 Codex 发给上游模型的 `input`，把这个 `input` 结构化成可视、可编辑的 transcript，再让一个副模型/上下文模型帮助用户编辑这份 transcript。

用户最终想要的是：

- 上下文地图展示的是 Codex `input` 数组，不是普通聊天摘要。
- transcript 是唯一业务真相。
- 代理核心只按最新重构方案走，不继承旧项目里混乱的 restore、override、多状态逻辑。
- 上下文工作台必须存在，它是“副模型作为 agent 操作 transcript”，也就是“AI 编辑 AI 的上下文”。

## 代理核心模型

代理应该只有一条干净路径：

1. 收到 Codex 请求体。
2. 用当前请求的 `input` 和 cursor 做 diff。
3. 把 pop/append 应用到 transcript。
4. 从 transcript 重组上游请求的 `input`。
5. 转发请求。
6. 解析响应，把 assistant output items 按设计追加回 cursor/transcript。

用户不想要“透传一套、重组一套”的分支。没有编辑时，从 transcript 重组出来的 input 应该自然等价于原始 input。

transcript 不是前端摘要，不是 override 层，也不是过滤后的历史记录。它就是上游 `input` 数组的结构化投影。

关键原话：

> "既然在input里，就要进入我的transcript啊，以role为节点"

> "我的这个是transcript一定要完全等于input数组，就算很多奇怪的元素我也宁愿去做兼容。"

## Compact 的最终意图

compact 只考虑 local compact，不要 remote compact。remote compact 看不到内容，不适合作为这个项目的主路径。

local compact 需要支持替换 Codex 原始压缩 prompt。compact 成功后，代理要立刻模拟 compact 后的状态，避免前端在 compact 完成到 Codex 下一轮请求之间出现空窗。

compact 成功后的模拟状态必须同时更新：

- `transcript`
- `codex_input_cursor`

为什么要模拟：如果不模拟，下一轮 Codex 请求确实可以把完整 input 全部 append 回来，逻辑上也能接上；但前端在这段间隔里拿不到 compact 后的数据，所以用户会看到空白或旧状态。

手动 compact 和自动 compact 的区别：

- 手动 compact：保留选中的若干 user message，再加 summary。
- 自动 compact：可能发生在 assistant 正在运行时，所以要排除当前还在运行的最后一段 user/assistant，让下一轮请求自然 append 回来。

关键原话：

> "compact的逻辑codex源码是有两个的，我们不要remotecompact，那个看不到内容，要走localcompact。然后我这里还做了压缩词替换的。"

> "每当compact的时候，代理向上游请求，然后上游返回一个压缩文本，此时我只需要把transcript清空，然后按照codex自己压缩上下文时会保留很多条user，我们transcript也这样去实现，然后把压缩文本塞到应该在的位置。然后再把cursor也改成模拟下一轮会发的前缀。"

> "你知道为什么我要做模拟吗，其实完全不做模拟也是可以的，每次压缩删除transcript和cursor。然后下一轮codex发请求了，当然就会全部append过来。这样很合理，但是就是要考虑前端展示，如果压缩完codex还没发下一次请求，前端拿不到数据所以我要做一个模拟。"

> "compact的时候transcript也是直接缓存模拟后的，就是只剩N条user加总结。"

## 前端意图

前端要跟随新后端方案设计。可以保留有价值的视觉样式和组件，但 API 和数据结构要以最新 transcript 模型为准。

上下文地图要能把 transcript 真实展示出来，包括工具调用、工具输出等结构化内容。工具不应该被混成难读的纯文本。

如果不知道某类工具或上下文项旧 UI 是怎么展示的，可以查 main；但查 main 是为了恢复成熟展示经验，不是为了恢复旧状态模型。

## 上下文模型的产品意图

上下文模型是一个副模型/agent，负责和用户对话，并通过工具操作主 transcript。当前可以先用同一个上游，但实现上最好留一个简单入口，方便以后接其他模型接口。

上下文模型要节省成本、减少调用：

- 不要为了简单任务频繁调用模型。
- 初始快照要轻，只给节点和 preview，不直接塞满完整 item。
- 模型需要细节时再调用 `get_node` 读取节点。
- 主流程是节点级编辑。
- item/event 级编辑保留，但不是主旋律。

关键原话：

> "副模型同一个上游。副模型的相关逻辑参考main分支的。什么系统提示词啊，快照啊什么的工具的话先不做，上下文模型这里暂时还是只做个骨架，我之后在思考怎么设计工具。"

> "为了成本考虑我现在觉得上下文模型不应该调很多次"

> "节点快照里有节点了，然后上下文模型可以有get_node（1,2,6）这种，就返回这些节点的transcript里的所有信息。"

> "write_node工具，就是可以指定节点然后写替换文本，可以这里默认模型是进行节点压缩，模型可以对这个节点的内容进行空替换=删除，压缩文本替换=压缩，这样一个工具就可以搞定。"

> "item级压缩这个功能尽量不用，不是主旋律了。"

## 上下文模型工具设计意图

工具应该围绕“当前工具循环开始时的快照编号”工作，而不是让模型主要依赖 UUID。

初始快照里只有 node 和一句 preview。模型从快照里不知道节点内部有多少 item，所以默认读取操作应该按节点读。

建议的工具心智模型：

- `get_node(nodes=[1,2,6])`：返回指定节点完整 transcript 信息。
- `write_node(...)`：一次完成节点级删除、替换、压缩、插入。
- item/event 工具：只在 assistant 节点内部需要更细操作时使用，优先级低于节点工具。

节点编辑工具一次应该支持：

- 删除连续或不连续范围，例如 `5-7`、`8`、`10-13`。
- 用压缩文本替换节点或节点范围。
- 只删除不插入，也就是纯删除。
- 在原始快照的某个节点后插入新节点。
- 用位置 `0` 表示插到最前，因为上下文地图没有节点 0。
- 插入时指定新 summary 节点的 role/content。

关键原话：

> "首先快照里只有node加一句preview，模型根本不知道节点里的item有几个，所以读的时候必须是只能读节点的。"

> "我想一个工具兼容多节点合并，或者去掉某些节点。"

> "我现在有一个思路就是插到初始节点的后面，比如我要替换5-7节点，那么就是删除5-7，然后插到节点四后面。删掉8，插到原始7后面，这里必须是以初始快照去判断的。"

> "这样可以一次工具兼容所有功能，删除替换，纯插入等等都可以。"

> "假如就要删掉5-7,8,10-13，那delete就填三个。然后插入不一定是三个都要，可能我只在7后面插，或者9后面插，5-7纯删除。又或者我直接就在别的地方插，这三都是纯删除。"

> "插入没必要插在最前面就可以写0。但是上下文地图是没有节点0的，刚好。"

## 上下文模型循环和 commit 语义

这里有过一次重要意图调整。

模型视角应该像实时修改：写工具执行后，工具返回更新后的快照，模型继续基于新快照推理。

实现层面可以在单个工具循环中维护一份临时草稿，并在循环结束后统一 commit，只要满足：

- 工具调用后，模型拿到的是更新后的快照。
- 不把旧项目的 restore/draft 复杂状态暴露到 API。
- 不在磁盘或前后端 payload 里制造很多份 transcript。
- 下一轮上下文模型对话不要长期保留上一轮工具输出。

上下文模型自己的对话历史应该很轻：

- 同一轮工具循环中，input 可以累计工具调用和工具输出，这样有利于缓存和连续推理。
- 当前循环结束后，下一轮只保留 user/assistant 文本对话。
- developer 信息在每轮新循环开始时重新生成。
- developer 里的快照只是初始说明，不应该在循环中被修改。

关键原话：

> "写节点的工具会返回更新后的新快照。模型就是应该会自己知道这是改完后啊。"

> "developer也是每次新循环才会更新，然后写节点的工具会返回更新后的新快照。"

> "上下文模型自己的历史只在当前工具循环轮会一直input里存在工具输出，只要当轮循环结束，下一轮对话的时候上下文模型拿到的只有user和assistant的文本对话。"

> "循环中input数组里的东西会一直累计，这样可以提高缓存，然后循环结束之后下一轮开始上下文才会把上一轮的工具输出啥的撇掉，只保留文本对话，继续下一轮。"

> "既然草稿没问题那就还是等循环结束后统一commit，但是对于上下文模型来说他是会觉得这个是实时在变的，因为工具每次返回新的transcript状态，当然实现里应该叫更改后的草稿，只是最后会统一commit，对吗"

## 启动和 shim 意图

Desktop 启动流程原来已经调好，不要轻易重造。

用户期望的流程：

1. 终端运行 `codex ctx desktop on` 开启代理。
2. 该命令像原来一样修改并备份 Codex config。
3. `off` 恢复 config。
4. `on` 可以先执行一次 off/reset。
5. 打开 Codex Desktop。
6. 在 Codex 输入框发送 `ctx`，打开上下文管理小窗。

关键原话：

> "首先是需要在终端输入on和off开启和关闭代理。然后打开codex，在输入框里发送ctx，然后打开小窗。"

> "你去看我之前的main的启动逻辑啊。这个不需要变。"

> "我之前的就是会codex ctx desktop on然后会将codex的config改一下，然后备份，off的话会恢复config。好像on也会先进行一次off。然后在on的时候用户打开codex就可以在开发1的时候输入ctx打开上下文管理面板。"

## 存储意图

代理状态不能只存在内存里。用户希望按不同对话落盘，放在用户级配置目录下。

最终方向：

- 使用类似 `~/.hash-context-codex/` 的用户级数据目录。
- 不同会话有不同文件夹。
- 存储文件要少、直观、能解释清楚。
- 不要又出现旧项目那种很多历史文件、tail 文件、restore 文件互相缠绕的状态。

关键原话：

> "现在对话还是储存在内存吗，不要在内存，我需要写磁盘的，就是以不同对话放到不同的文件夹下，我们这个项目使用的时候会在用户电脑上出现配置文件夹吗，.codedx这种，把对话文件夹放里面"

> "怎么那么多储存文件，我刚刚发了一句话，没有transcript，只有tail。不对啊，那么多文件都是干什么的，当时重构计划里就几个储存的东西啊"

## 排障纪律

用户反馈 bug 时，必须严格对应现象排查。没有明确证据或复现路径，不要直接给“修复方案”或乱改。

应该遵守：

- 启动慢就查慢在哪里，不要直接加 timeout。
- 白屏就看前端错误、DevTools、接口数据形状。
- Codex 一直思考就查请求、响应、SSE 流。
- 手动页 session not found 就查 session 创建和路由假设。
- main 里已有成熟兼容逻辑时，先查 main，再搬必要的外围逻辑。

关键原话：

> "你以后有好的排查方案需要我去做就早点说，早说可以打开这个开发者工具不就行了。"

> "不是，你这启动额太久了啊，你去查慢的原因啊，而不是直接加时间"

> "我是让你去检查有没有bug之类的"

> "什么鬼啊，我都说了main里面全都做好了的，我当时只需改一个方案，别的所有都是可以去main里找到逻辑的，你去找啊，你把main怎么处理这个的搬过来。"

## 沟通偏好

用户是偏技术的产品经理。解释时要优先讲逻辑、数据流、产品含义，必要时再解释函数或代码点。

如果用户的提示词比较乱，要先帮用户梳理意图，确认方向，再动手。信息不足时要问，不要自己猜着浪费时间。

关键原话：

> "我是偏技术的产品经理，你不能经常做完任务给我说一堆很复杂代码里的函数，要重点讲逻辑，必要的术语解释是什么，函数引用解释功能。"

> "遇到信息缺失可以随时停下来问我，不要自己浪费时间去搜，即使你可能找得到。"

> "我的提示词乱的时候，你需要主动先帮我梳理让我确定我的意思已经传达，再干活。"

> "如果无法确认某个问题点有较大把握（即没有十足的证据或明确的复现路径），就不要强行给出“修复方案”或“结论”。"

## 后续改代码前的检查清单

每次改代理相关代码前，先问：

- 是否符合 `docs/proxy-design.md`？
- transcript 是否仍然等价于 input？
- 请求处理是否仍然只有一条路径？
- compact 是否只走 local compact？
- compact 后 transcript 和 cursor 是否都按设计更新？
- 代码里有没有重新出现 `override`、restore、pending restore、旧 draft/revision API？
- 前端是否读取新的 transcript 结构，而不是旧 payload？
- 落盘是否简单、按会话、能解释清楚？
- 上下文模型是否基于快照和工具工作，而不是长期堆积工具输出？
- bug 是否已经复现或追踪到证据，再开始修改？

如果答案不明确，先停下来问用户。
