## 1. 背景与目标

### 1.1 背景

Codex CLI 和 Codex Desktop 已经成为很多开发者处理代码任务的主要入口。它们适合持续对话和多轮执行，但在长会话里，上下文会持续累积：工具输出、搜索结果、失败尝试、过期假设、重复讨论和中间结论都会被带入后续请求。

当会话变长时，用户会遇到三个问题：

1. **看不见**：用户不知道下一轮 Codex 实际会带着哪些历史继续工作。
2. **改不了**：用户无法在下一轮请求前精确删除、压缩或修正历史上下文。
3. **不放心**：自动压缩发生时机和保留内容不透明，用户难以判断关键结论是否被保留。

Codex Context Proxy 希望在不替代 Codex 的前提下，为 Codex 增加一层本地、可视化、可编辑、可恢复的上下文工作台。

### 1.2 产品定位

Codex Context Proxy 不是新的 IDE，也不是新的聊天工具，而是 Codex 前面的一层本地上下文管理助手。

一句话定义：

> 让 Codex 的上下文从“只能追加的聊天历史”，变成“下一轮请求前可查看、可编辑、可恢复的工作材料”。

用户仍然使用原来的 Codex CLI 或 Codex Desktop；当需要管理上下文时，通过小窗打开上下文工作台，对当前会话进行查看、压缩、删除、替换和恢复。

### 1.3 产品目标

| 目标 | 说明 |
|---|---|
| 可见 | 用户能看到当前 Codex 会话的结构化上下文，知道模型下一轮大概会参考什么 |
| 可控 | 用户能删除、压缩、替换上下文节点或节点内内容 |
| 可辅助 | 用户能让独立的上下文工作台模型帮助诊断和整理上下文 |
| 可恢复 | 每次有效编辑形成 revision，用户能恢复到历史版本，并撤回本次恢复 |
| 不打断 | 未编辑时尽量保持原生 Codex 体验；编辑后只影响后续请求 |
| 安全稳妥 | 运行中、压缩中等不稳定状态下禁止危险编辑 |

### 1.4 第一版成功标准

1. 用户可以通过 Codex CLI 正常进入代理后的 Codex 使用流程。
2. 未编辑上下文时，普通 Codex 对话体验接近原生。
3. 用户可以打开上下文小窗并看到当前 active Codex session。
4. 用户可以在小窗中删除、压缩或替换上下文，并让编辑结果影响下一轮 Codex 请求。
5. 用户可以恢复到任意历史 revision，并撤回本次恢复。
6. 当前 turn 正在运行或系统正在压缩上下文时，工作台保持只读，避免破坏上下文一致性。
7. CLI shim 可以开启、关闭、查看状态和卸载，不破坏官方 Codex 命令。
8. Codex Desktop 接入保持实验能力，并提供状态检查和恢复入口。

## 2. 用户画像与核心场景

### 2.1 目标用户

| 用户类型 | 特征 | 主要诉求 |
|---|---|---|
| 高频 Codex CLI 用户 | 长时间在终端里使用 Codex 做开发 | 不想换工具，但希望能管理长上下文 |
| 长任务开发者 | 一次任务跨多个小时、多轮调试、多次工具调用 | 清理噪声、保留关键结论、减少误导 |
| 上下文敏感用户 | 对模型是否读到错误历史很敏感 | 精确修正错误假设和过期信息 |
| Codex Desktop 用户 | 使用桌面端做代码任务 | 希望桌面端也能获得上下文工作台能力 |
| ChatGPT 订阅或 API 用户 | 使用不同登录方式访问 Codex | 希望无需额外配置就能复用现有 Codex 使用方式 |

### 2.2 用户痛点

1. **长会话噪声过多**：命令输出、失败尝试、调试日志和重复解释占用大量上下文。
2. **错误假设会持续污染后续回答**：一旦模型早期形成错误判断，用户只能靠追加消息纠正，不能直接改掉历史。
3. **话题切换成本高**：用户想保留结论，但不想带上完整过程。
4. **自动 compact 不透明**：用户不知道什么时候压缩、压缩后保留了什么、丢了什么。
5. **上下文编辑缺少回滚能力**：即使用户能手动整理，也需要知道可以恢复，才敢放心编辑。

### 2.3 核心场景

#### 场景 A：长任务中清理噪声

用户让 Codex 连续修复一个复杂问题，中途产生大量工具输出、失败命令和过时猜测。继续对话前，用户打开上下文工作台，发现某几个 assistant 节点很重，于是把它们压缩成“问题、尝试、结论、剩余风险”的摘要。

**用户收益**：后续请求保留关键结论，减少历史噪声影响。

#### 场景 B：话题切换前保留关键结论

用户从 bug 修复切到架构讨论，希望保留刚才的修复结果，但不想把完整调试过程带入新话题。用户选中相关节点，将它们压缩为一段简洁上下文。

**用户收益**：话题切换更干净，Codex 不容易被旧过程带偏。

#### 场景 C：修正误导信息

Codex 在早期会话中形成了一个错误假设。用户在上下文地图中定位该节点，把错误内容替换成正确说明，再继续下一轮提问。

**用户收益**：从源头修正历史上下文，而不是不断追加“你刚才说错了”。

#### 场景 D：运行中查看但不编辑

Codex 正在生成回答或执行工具调用时，用户打开小窗查看当前上下文。此时工作台展示运行状态，但编辑入口不可用。

**用户收益**：可以观察正在发生什么，但不会破坏尚未稳定的上下文。

#### 场景 E：误编辑后恢复

用户压缩或删除了过多内容，发现下一轮 Codex 表现不理想。用户进入恢复页，选择之前的 revision 恢复，并可撤回这次恢复。

**用户收益**：上下文编辑有安全网，用户敢于主动整理。

#### 场景 F：Codex Desktop 实验接入

用户希望 Desktop 也能使用同一套上下文能力。用户开启 Desktop 代理后，用 fresh chat 测试；如果不符合预期，可以查看状态、修复配置或关闭恢复。

**用户收益**：桌面端获得同样能力，但风险有明确提示和回退路径。

## 3. 产品范围

### 3.1 第一版范围

| 模块 | 用户心智 | 第一版范围 |
|---|---|---|
| Codex CLI 接入 | 继续使用原来的 Codex | 支持通过命令或 shim 进入本地代理后的 Codex |
| Codex Desktop 实验接入 | 桌面端也可试用 | 支持开启、关闭、状态检查、探测和修复 |
| 上下文小窗 | 查看和整理上下文的入口 | 可隐藏启动，可通过 `context` / `ctx` 唤起 |
| 上下文地图 | 当前会话结构化视图 | 展示用户轮次、助手轮次、工具内容和 token 负载 |
| 建议页 | 快速判断哪里需要清理 | 展示 token、工具 token、重节点等建议信息 |
| 手动页 | 让上下文模型协助编辑 | 支持独立对话、选择节点、流式输出和停止 |
| 编辑能力 | 直接改变下一轮上下文 | 支持删除、压缩、替换节点或节点内内容 |
| 恢复能力 | 编辑安全网 | 支持 revision、恢复和撤回恢复 |
| 设置页 | 管理工作台偏好 | 支持上下文模型、推理强度、token 阈值、工具目录等设置 |
| 用量展示 | 理解上下文成本 | 展示当前会话用量，并支持清空统计 |

### 3.2 非目标

第一版不做以下事情：

- 不重做 Codex CLI UI。
- 不直接修改 Codex 源码。
- 不把 Codex session 文件作为主数据源。
- 不保证所有模型服务接入方式都能通过代理链路工作。
- 不把 Desktop 支持定义为和 CLI 一样稳定的主路径。
- 不做全自动上下文维护模式。
- 不在 PRD 中定义具体协议字段、请求体改写方式、源码路径和实现算法；这些内容归属技术实现文档。

### 3.3 产品边界与依赖

| 类型 | 说明 | 产品要求 |
|---|---|---|
| Codex 依赖 | 产品依赖 Codex 的 CLI / Desktop 使用入口 | Codex 原有体验应尽量不被破坏 |
| 本地代理依赖 | 上下文能力发生在本地请求边界 | 未编辑时透明，编辑后以后续请求生效 |
| 上下文压缩依赖 | Codex 可能触发自动或手动 compact | 压缩结果必须尊重用户已编辑的上下文 |
| 授权依赖 | 用户可能通过 ChatGPT 订阅或 API key 使用 Codex | 工作台应尽量复用现有 Codex 授权，失败时给出明确提示 |
| Desktop 依赖 | Desktop 接入需要改动本地 Codex 配置 | 必须有实验标记、备份、状态检查和关闭恢复能力 |

## 4. 产品结构与用户路径

### 4.1 模块关系

| 模块 | 面向用户的解释 | 主要产品职责 |
|---|---|---|
| Codex CLI / Desktop | 用户原本使用的工作入口 | 承载真实代码任务和对话体验 |
| 本地上下文代理 | Codex 前面的上下文闸门 | 捕获上下文、应用用户编辑、同步压缩结果 |
| Electron 小窗 | 上下文工作台窗口 | 展示当前会话，承载查看、编辑、恢复、设置 |
| 上下文工作台 | 整理上下文的功能区 | 建议、手动整理、恢复、模型设置 |
| 上下文地图 | 会话结构化视图 | 帮助用户定位重节点、错误节点和可压缩内容 |

### 4.2 CLI 主路径

1. 用户启动 Codex Context Proxy。
2. 系统启动本地代理和上下文小窗，随后进入 Codex CLI。
3. 用户像平时一样使用 Codex。
4. 用户需要整理上下文时，在 Codex 中输入 `context`、`ctx`、`/context` 或 `/ctx`。
5. 系统打开小窗，并拦截该控制命令，不发送给模型。
6. 用户查看上下文地图，进行删除、压缩、替换或恢复。
7. 用户应用编辑。
8. 下一轮 Codex 请求使用编辑后的上下文继续工作。

### 4.3 Desktop 实验路径

1. 用户开启 Desktop 代理能力。
2. 系统提示该路径为实验能力，建议使用 fresh chat 测试。
3. 系统写入并管理必要配置，同时保留恢复能力。
4. 用户在 Desktop 中正常发起新对话。
5. 用户通过上下文小窗查看和管理当前会话。
6. 如果 Desktop 行为异常，用户可查看状态、执行修复或关闭代理恢复配置。

### 4.4 上下文编辑路径

1. 用户在上下文地图中浏览当前会话。
2. 用户通过节点选择、展开、搜索或建议页定位要处理的上下文。
3. 用户选择手动编辑或让上下文模型辅助编辑。
4. 用户确认编辑结果。
5. 系统生成 revision。
6. 编辑结果成为后续 Codex 请求的上下文来源。

### 4.5 恢复路径

1. 用户进入恢复页。
2. 用户查看历史 revision 的时间、摘要和变更类型。
3. 用户选择目标 revision 恢复。
4. 系统切换当前上下文和手动页历史。
5. 用户可以继续使用恢复后的上下文，也可以撤回本次恢复。

## 6. 功能详细设计

### 6.1 启动与接入

#### 用户目标

用户希望尽量不改变原有 Codex 使用习惯，只是在需要时获得上下文管理能力。

#### 功能说明

- 用户可以通过项目提供的一键入口启动 Codex Context Proxy。
- 系统自动启动必要的本地服务和上下文小窗。
- 用户可通过 CLI shim 让日常 `codex` 命令进入代理链路。
- 用户可随时关闭 shim，让 `codex` 回到官方 Codex。
- 高级用户可以指定真实 Codex 命令和本地 host / port。

#### 异常情况

| 情况 | 产品表现 |
|---|---|
| 本地服务启动失败 | 提示失败模块和日志位置，不进入误导性成功状态 |
| 找不到真实 Codex | 提示用户配置真实 Codex 命令 |
| 端口占用 | 尝试处理受管理的旧进程；无法处理时给出明确提示 |
| shim 状态异常 | status 能展示当前状态，并提供关闭或卸载路径 |

#### 验收标准

- 用户可以完成启动、普通对话、打开小窗、关闭代理的完整闭环。
- shim 开启和关闭后，`codex` 命令行为符合用户预期。

### 6.2 Codex Desktop 实验支持

#### 用户目标

Desktop 用户希望在不长期破坏本地配置的前提下，试用上下文工作台能力。

#### 功能说明

- Desktop 接入必须明确标记为实验能力。
- 开启前提示用户建议使用 fresh chat 测试。
- 系统提供开启、关闭、状态检查、环境探测和修复能力。
- 修改用户配置前必须保留备份。
- 关闭后应恢复受管理的配置内容。

#### 异常情况

| 情况 | 产品表现 |
|---|---|
| 用户配置格式异常 | 提示风险，并引导使用 repair 或手动检查 |
| Desktop 未走代理 | status / probe 给出可能原因 |
| 用户希望退出实验 | off 能移除受管理配置并停止相关服务 |

#### 验收标准

- Desktop 路径不会被描述为稳定主路径。
- 用户能明确知道如何开启、测试、检查和回退。

### 6.3 上下文小窗

#### 用户目标

用户希望在不中断 Codex 主体验的情况下，随时打开一个辅助窗口查看上下文。

#### 功能说明

- 小窗默认隐藏启动。
- 用户可通过 `context`、`ctx`、`/context`、`/ctx` 唤起。
- 控制命令只用于打开小窗，不应进入模型上下文。
- 小窗优先展示当前 active session。
- 无真实 session 时可以展示 demo 或空状态说明，避免用户误以为功能损坏。

#### 状态表现

| 状态 | 用户侧表现 |
|---|---|
| 无 session | 展示引导或 demo，提示先通过代理发起 Codex 对话 |
| 有 active session | 展示该 session 的上下文地图和工作台 |
| Codex 正在运行 | 展示运行状态，编辑能力只读 |
| session 出错 | 展示错误提示和可查看的已有上下文 |

#### 验收标准

- 用户输入控制命令后，小窗能稳定显示。
- 控制命令不会污染 Codex 对话。

### 6.4 上下文地图

#### 用户目标

用户希望快速理解当前会话里有哪些内容、哪些内容最重、哪些内容值得清理。

#### 功能说明

- 顶层按用户轮次和助手轮次展示。
- 助手节点内部保留工具调用、工具结果、推理摘要和其他模型返回内容。
- 支持展开、折叠、选择、多选、拖选和 minimap。
- 支持用颜色表达 token 负载。
- 内部系统性前缀内容不参与用户编辑编号，避免误删关键上下文。

#### 验收标准

- 长会话下用户仍能定位某一轮或某个重节点。
- 用户能区分普通对话内容、工具输出和内部不可编辑内容。

### 6.5 建议页

#### 用户目标

用户希望不手动逐个节点检查，也能知道当前上下文最值得清理的位置。

#### 功能说明

- 展示当前 session 的 token 概览。
- 展示工具 token 占比和重节点列表。
- 标识可能适合压缩或删除的节点。
- 第一版建议页以“辅助判断”为主，不默认自动应用。

#### 后续方向

- P2 支持一键采纳建议。
- P2 支持用户授权后的自动维护模式。

#### 验收标准

- 用户打开建议页后，能明确知道哪些节点最重、为什么值得处理。

### 6.6 手动工作台

#### 用户目标

用户希望让一个独立的上下文模型帮自己分析、压缩、替换或删除上下文，而不是完全手工操作。

#### 功能说明

- 手动页支持独立对话，不影响 Codex 主会话。
- 用户选中上下文节点后，输入框显示节点 chips。
- 上下文模型可以读取选中节点详情，并执行删除、压缩、替换等操作。
- 模型请求支持流式输出。
- 用户可以停止正在运行的工作台请求。
- 每轮模型编辑先在工作快照中完成，结束后统一提交。

#### 异常情况

| 情况 | 产品表现 |
|---|---|
| 未选择节点但要求编辑 | 提示需要选择目标或让模型先查找 |
| 模型请求过长 | 用户可以停止请求 |
| 模型没有实际修改 | 不生成 revision |
| Codex 主会话正在运行 | 工作台只读或禁止提交编辑 |

#### 验收标准

- 工作台模型对话不会污染 Codex 主上下文。
- 有效编辑完成后生成 revision，并同步影响后续 Codex 请求。

### 6.7 编辑能力

#### 用户目标

用户希望精准控制下一轮 Codex 可以看到的上下文。

#### 功能说明

- 支持删除一个或多个上下文节点。
- 支持将一个或多个节点压缩为摘要。
- 支持删除、替换、压缩节点内的单个或多个内容项。
- 支持批量查找和批量编辑。
- 应用编辑后，session 进入已编辑状态。
- 用户可 reset 回到未编辑状态。

#### 编辑原则

1. 编辑只影响后续请求，不回写已经展示过的 Codex 历史 UI。
2. 用户删除或替换的内容，不应在下一轮被系统重新带回。
3. 正在运行或压缩中的 session 不允许提交编辑。
4. 每次有效编辑必须有可恢复记录。

#### 验收标准

- 删除、压缩、替换后的内容在下一轮 Codex 中生效。
- 用户能看出当前 session 是否处于已编辑状态。

### 6.8 恢复与 revision

#### 用户目标

用户希望上下文编辑有安全网，可以放心尝试整理。

#### 功能说明

- 每次有效修改生成一个 revision。
- revision 记录时间、摘要、变更类型和完整上下文快照。
- 用户可恢复到任意 revision。
- 恢复会同步手动页历史，让用户看到当时的上下文工作过程。
- 用户可撤回本次恢复，回到恢复前状态。

#### 异常情况

| 情况 | 产品表现 |
|---|---|
| 目标 revision 不存在 | 提示恢复失败，不改变当前上下文 |
| 当前 session 正在运行 | 禁止恢复 |
| 恢复后想反悔 | 提供撤回本次恢复入口 |

#### 验收标准

- 用户能从恢复页明确识别每个 revision。
- 恢复和撤回恢复均能影响后续 Codex 请求上下文。

### 6.9 Compact 与长上下文

#### 用户目标

用户希望系统压缩长上下文时，不绕过自己已经做过的编辑。

#### 功能说明

- 当 Codex 或系统进入上下文压缩流程时，工作台显示压缩中状态。
- 压缩期间禁止危险编辑。
- 如果用户已编辑上下文，压缩应基于编辑后的上下文。
- 压缩完成后，工作台展示压缩后的上下文结果。
- 压缩失败时，不应破坏当前可用上下文。

#### 验收标准

- 用户已删除或替换的内容不会因为压缩被重新引入。
- 压缩中用户无法提交编辑、恢复或 reset。

### 6.10 用量与设置

#### 用户目标

用户希望理解上下文整理带来的成本，并能调整工作台模型和显示偏好。

#### 功能说明

- 展示当前 session 的主会话、上下文工作台、压缩相关用量。
- 支持清空当前 session 用量统计。
- 设置页支持选择上下文模型来源、模型和推理强度。
- 设置页展示可用工具目录。
- 设置页支持 token 阈值配置。

#### 验收标准

- 用户能看到当前会话各类用量。
- 用户修改设置后，新设置能影响后续工作台请求或显示。

## 7. 权限、安全与风险边界

### 7.1 本地数据边界

- 产品以本地代理和本地工作台为主，不把上下文管理定义为云端服务。
- 用户的会话上下文、revision、设置和日志应默认保存在本地。
- 日志中如涉及授权、请求头或敏感内容，应避免直接展示完整敏感信息。

### 7.2 授权复用边界

- 工作台应尽量复用用户已有 Codex 授权，减少额外登录和配置。
- 如果授权尚未捕获或不可用，应明确提示用户先通过 Codex 主路径发送普通消息。
- 授权失败不应导致上下文地图和本地编辑能力不可用。

### 7.3 用户编辑风险

| 风险 | 产品处理 |
|---|---|
| 用户误删关键上下文 | revision 和撤回恢复提供安全网 |
| 用户在运行中编辑导致状态错乱 | 运行中和压缩中只读 |
| 用户不理解编辑是否生效 | 明确展示 session 状态和下一轮生效规则 |
| 用户压缩过度导致信息丢失 | 建议保留恢复入口，并在压缩摘要中突出关键信息 |

### 7.4 Desktop 实验风险

Desktop 接入对本地 Codex 配置更敏感，因此产品必须：

1. 明确标记为实验功能。
2. 开启前提示 fresh chat 测试。
3. 修改前备份。
4. 提供 status / probe / repair / off。
5. 不把 Desktop 作为第一版最稳定主路径承诺。

## 8. 数据指标与质量指标

### 8.1 使用指标

| 指标 | 目的 |
|---|---|
| 小窗打开次数 | 判断用户是否主动使用上下文管理 |
| 每个 session 的编辑次数 | 判断上下文编辑是否形成真实需求 |
| 删除 / 压缩 / 替换操作占比 | 判断最常用整理方式 |
| revision 恢复次数 | 判断用户误编辑或回滚需求强度 |
| Desktop 开启与关闭次数 | 判断实验接入可用性和稳定性 |

### 8.2 质量指标

| 指标 | 目标 |
|---|---|
| 未编辑普通对话成功率 | 接近原生 Codex |
| 编辑后生效率 | 删除、替换、压缩能稳定影响下一轮请求 |
| 运行中误编辑率 | 应为 0 |
| 压缩后上下文污染率 | 已删除或替换内容不应被重新引入 |
| 工作台内部请求污染率 | 应为 0 |

### 8.3 体验指标

| 指标 | 目标 |
|---|---|
| 小窗打开耗时 | 用户可接受，不明显阻塞主流程 |
| 长会话上下文地图可操作性 | 节点展开、选择、滚动不卡死 |
| 编辑完成反馈 | 用户能明确知道是否已应用、是否已生成 revision |
| 错误提示清晰度 | 用户知道失败原因和下一步该做什么 |

## 9. 验收计划

### 9.1 P0 主流程验收

1. 启动 Codex Context Proxy。
2. 在 Codex 中发送普通消息，确认响应正常。
3. 输入 `context` 或 `ctx`，确认小窗打开且控制命令不进入模型。
4. 在小窗中查看 active session 上下文地图。
5. 删除或压缩一个历史节点。
6. 应用编辑并继续下一轮 Codex 对话。
7. 确认下一轮不再依赖被删除或替换的内容。
8. 在恢复页恢复到编辑前 revision。
9. 确认恢复后的上下文影响下一轮对话。
10. 在 Codex 正在运行时打开小窗，确认编辑能力只读。

### 9.2 P1 验收

1. 开启和关闭 CLI shim，确认 `codex` 行为符合状态。
2. 使用 Desktop 实验入口完成开启、状态检查和关闭恢复。
3. 在建议页查看 token 概览和重节点。
4. 在手动页选择节点，让上下文模型进行压缩或替换。
5. 停止一次正在运行的上下文模型请求。
6. 修改上下文模型设置和 token 阈值，确认后续请求或展示生效。
7. 查看并清空当前 session 用量统计。

### 9.3 回归验收

1. 未编辑上下文时，普通 Codex 对话体验不应明显变化。
2. 工作台内部模型对话不应进入 Codex 主上下文。
3. reset 后应回到未编辑状态。
4. 压缩中和运行中不允许提交编辑、恢复或 reset。
5. 代理或工作台失败时，已有上下文和 revision 不应被破坏。

## 10. 里程碑

| 阶段 | 目标 | 交付标准 |
|---|---|---|
| M0 | 本地代理、小窗、CLI wrapper 可运行 | 用户能进入代理后的 Codex，并打开上下文小窗 |
| M1 | 捕获 session 并展示上下文地图 | 用户能看到 active session 的结构化上下文 |
| M2 | 编辑后下一轮生效 | 删除、压缩、替换能影响后续 Codex 请求 |
| M3 | 恢复和 revision 完整闭环 | 用户能恢复任意 revision，并撤回本次恢复 |
| M4 | 长上下文压缩接管 | 压缩流程尊重用户已编辑上下文，压缩中只读 |
| M5 | CLI shim 和 Windows 打包 | 用户可安装、开启、关闭和卸载，不破坏官方 Codex |
| M6 | Desktop 实验支持 | 用户可试用 Desktop 接入，并可检查和恢复配置 |
| M7 | 智能建议和自动维护 | 支持一键建议应用，并探索授权后的自动维护模式 |

## 11. 开放问题

| 编号 | 问题 | 影响 | 当前原则 |
|---|---|---|---|
| Q-01 | Codex 上游行为变化可能影响代理稳定性 | 普通对话、上下文编辑或压缩可能失效 | 技术实现文档维护协议细节和测试覆盖 |
| Q-02 | Desktop 接入依赖用户本地配置 | 配置异常可能导致 Desktop 代理不可用 | 保持实验标记、备份和恢复能力 |
| Q-03 | 用户过度压缩可能降低模型后续质量 | Codex 可能缺少关键细节 | 保留 revision，鼓励摘要包含结论、证据和风险 |
| Q-04 | 长会话 revision 可能带来本地存储增长 | 本地数据目录持续变大 | 后续评估清理策略或增量存储 |
| Q-05 | 工作台授权复用可能在首次使用时不可用 | 用户无法立即使用上下文模型辅助能力 | 提示先通过代理发送普通 Codex 消息 |
| Q-06 | 全自动上下文维护可能误删重要内容 | 用户信任风险高 | P2 阶段必须先提供预览、确认和可回滚机制 |
